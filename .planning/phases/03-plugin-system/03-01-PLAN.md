**Read FIRST:** `.planning/phases/03-plugin-system/03-RESEARCH.md` section "Pattern 3: Conflict Detection Algorithm" (lines ~270-373) for the full class design including all four internal Maps, the register method flow, and accessor methods. `.planning/phases/03-plugin-system/03-CONTEXT.md` decisions D-02 through D-06.

**DO:**

Create `src/plugin-registry.ts` with:

1. **Internal state (4 Maps):**
   - `contentPlugins = new Map<string, ContentPlugin>()` — name → plugin
   - `uiPlugins = new Map<string, UIPlugin>()` — name → plugin
   - `tokenTypeOwners = new Map<string, string>()` — TokenDef.type → owning plugin name
   - `slotAssignments = new Map<string, Map<number, string>>()` — slot name → (priority → plugin name)

2. **Semver validation (per D-02):**
   Private method `validateVersion(version: string): void`. Uses regex `/^\d+\.\d+\.\d+(-[\w.]+)?$/`. Throws `Error` with message `Invalid version "${version}" for plugin "${name}": must match semver format X.Y.Z or X.Y.Z-prerelease`.

3. **`register(manifest: PluginManifest): void` method flow:**
   a. Call `validateVersion(manifest.version)` — throw if invalid.
   b. Check name replacement: if a plugin with the same name exists in any category, call its `onDestroy?.()` and remove old ownership. Use `switch(manifest.kind)` to determine which Map to check.
   c. Category-specific registration via switch:
      - `case 'content'`: For each `def` in `manifest.tokens`, check `tokenTypeOwners.get(def.type)`. If a DIFFERENT plugin name owns it, throw `Error` with: `Plugin conflict: "${manifest.name}" declares token type "${def.type}", but "${owner}" already owns it. Each token type may only be registered by one content plugin.`. Then set `tokenTypeOwners.set(def.type, manifest.name)`. Finally `contentPlugins.set(manifest.name, manifest)`.
      - `case 'ui'`: For each `slot` in `manifest.slots`, get or create the priority map for that slot. Use `manifest.priority ?? 0`. If a DIFFERENT plugin name already has that priority in that slot, throw `Error` with: `UI plugin conflict: "${manifest.name}" claims slot "${slot}" with priority ${priority}, but "${existing}" already claims it with the same priority. Change one plugin's priority to resolve.`. Then set the priority entry and `uiPlugins.set(manifest.name, manifest)`.
      - `case 'storage'`: `contentPlugins` → no, use a separate Map. Actually, `StoragePlugin` has no conflict detection (no tokens/slots). Just set in a `storagePlugins` Map. Add a `storagePlugins` Map.
   d. Call `manifest.onInit?.()` AFTER successful registration.

4. **Name replacement helper (per D-06):**
   Private method `removeByName(name: string): void` that:
   - Checks all three Maps for a plugin with matching name
   - Calls `plugin.onDestroy?.()`
   - Removes from the appropriate Map
   - For content plugins: removes all token types from `tokenTypeOwners` where the owner matches
   - For UI plugins: removes all slot+priority entries from `slotAssignments` where the owner matches

5. **Accessors:**
   - `allContentPlugins(): ContentPlugin[]` — returns `Array.from(this.contentPlugins.values())`
   - `allTokenDefs(): TokenDef[]` — returns `this.allContentPlugins().flatMap(p => p.tokens)`
   - `getContentPluginByType(type: string): ContentPlugin | undefined` — looks up `tokenTypeOwners.get(type)` then `contentPlugins.get(ownerName)`
   - `getPlugin(name: string): PluginManifest | undefined` — checks all three Maps
   - `getAllPlugins(): PluginManifest[]` — concatenates values from all three Maps
   - `clear(): void` — clears all Maps (no lifecycle calls — caller manages teardown)

6. Run `npm run typecheck` to verify the class compiles.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Plugin author → PluginRegistry.register() | Manifest data crosses trust boundary (plugin author controls name, version, tokens) |
| PluginRegistry → Downstream consumers (tokenizer/renderer) | Plugin data is consumed by tokenizer/renderer — types enforce safe dispatch |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Tampering | PluginRegistry.register() version validation | mitigate | Semver regex validation at registration time catches malformed version strings per D-02; invalid versions throw before plugin is stored |
| T-03-02 | Denial of Service | PluginRegistry.register() token registration | accept | Plugin token patterns are authored by the developer integrating the library, not by end users. Patterns are trusted input. No ReDoS risk at this phase. |
| T-03-03 | Information Disclosure | Conflict detection error messages | accept | Error messages include plugin names and token types — these are developer-authored strings, not user data. No PII exposure risk. |
| T-03-04 | Elevation of Privilege | PluginRegistry name replacement | mitigate | Plugin replacement only affects plugins with matching name — developers control which plugins they register. No cross-plugin privilege escalation exists since plugins have no access to each other's state. |
</threat_model>
