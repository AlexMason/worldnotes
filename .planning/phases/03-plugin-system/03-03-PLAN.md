<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| EditorBuilder.use() → PluginRegistry.register() | Manifest data crosses trust boundary — validated by registry |
| Plugin lifecycle hooks → Editor internals | Plugin code runs in editor context during onInit/onUpdate/onDestroy |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-07 | Denial of Service | editor-lifecycle.ts destroy() | mitigate | onDestroy calls wrapped in try/catch per Pitfall 3 — one failing plugin cannot block teardown of others |
| T-03-08 | Denial of Service | editor-lifecycle.ts onUpdate | accept | onUpdate is sync and called after every render. If a plugin's onUpdate is slow, typing feels laggy. This is the plugin author's responsibility — documented in API docs. |
| T-03-09 | Information Disclosure | Plugin error messages in console | accept | console.error in onDestroy catch block logs plugin name and error — these are developer-authored plugin names, not user data. Acceptable for debugging. |
| T-03-10 | Elevation of Privilege | PluginRegistry.clear() | mitigate | clear() does NOT call onDestroy — intentional design choice documented in code. EditorInstance.destroy() is the proper teardown path that calls onDestroy. clear() is for pre-mount reconfiguration. |
</threat_model>
