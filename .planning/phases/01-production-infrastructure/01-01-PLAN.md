<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| npm registry → project | Supply chain — 10 dev dependencies installed from npm. Malicious packages could inject code into CI or local development |
| GitHub Actions runner → repository | CI has read access to source and write access to status checks. No secrets needed in Phase 1 |
| ESLint config → source code | Static analysis only — no runtime consequences. Misconfiguration could miss security issues |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Tampering | npm install (supply chain) | accept | Lockfile pins exact versions. `npm ci` in CI uses lockfile for reproducible installs. No runtime deps — dev-only risk |
| T-01-02 | Tampering | ESLint config disabling security rules | mitigate | The recommended + stylistic presets include security-relevant rules. Do not override them without explicit justification. Document any rule exceptions in-plan |
| T-01-03 | Information Disclosure | CI workflow (no secrets) | accept | No secrets, tokens, or credentials needed in Phase 1 CI. All commands run on public source code |
| T-01-04 | Elevation of Privilege | GitHub Actions workflow injection | mitigate | Use `actions/checkout@v6` and `actions/setup-node@v6` pinned to major versions. No user-controlled input interpolated into shell commands. `npm ci` (not install) avoids scripts from untrusted packages |
</threat_model>
